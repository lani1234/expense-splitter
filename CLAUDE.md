## Commands

### Backend (Maven / Spring Boot)
```bash
mvn spring-boot:run       # Run the application
mvn test                  # Run all tests
mvn test -Dtest=ClassName # Run a single test class
mvn clean install         # Build (skip tests: add -DskipTests)
```

### Frontend (Vite / React — in `frontend/`)
```bash
cd frontend
npm install
npm run dev    # Dev server (proxies /api → http://localhost:8080)
npm run build  # Production build (tsc + vite)
```

### Database
PostgreSQL is required locally. Copy `src/main/resources/application.properties.example` to `application.properties` and fill in credentials. Flyway migrations run automatically on startup from `src/main/resources/db/migration/`.

## Architecture

### Backend
Spring Boot 4.0.3 / Java 21 REST API with a standard three-layer architecture:

- **Controllers** (`com.expensesplitter.controller`) — REST endpoints; all responses wrapped in `ApiResponse<T>`. Three controllers: `TemplateController` (`/api/templates`), `InstanceController` (`/api/instances`), `ParticipantEntryAmountController` (`/api/participant-entry-amounts`). `GlobalExceptionHandler` maps `ValidationException`, `ResourceNotFoundException`, and `ApiException` to HTTP responses.
- **Services** (`com.expensesplitter.service`) — Business logic; all methods are `@Transactional`. `InstanceService` is the most complex, handling split calculation across the three split modes.
- **Repositories** (`com.expensesplitter.repository`) — Spring Data JPA extending `JpaRepository`.
- **Entities** (`com.expensesplitter.entity`) — JPA entities with Lombok; UUID PKs generated by PostgreSQL (`gen_random_uuid()`).
- **DTOs** (`com.expensesplitter.dto`) — Java 21 records with static `from(Entity)` factory methods. Controllers map entities → DTOs at the boundary (never return raw entities). This is required because JPA entities use `FetchType.LAZY` for `@ManyToOne` relations, so serializing them directly produces nested objects instead of flat UUID fields the frontend expects. DTO list: `TemplateInstanceResponse`, `InstanceFieldValueResponse`, `TemplateFieldResponse`, `TemplateParticipantResponse`, `SplitRuleResponse`, `SplitRuleAllocationResponse`, `ParticipantEntryAmountResponse`.

### Domain Model
The core domain:
- **Template** → defines the structure: participants (`TemplateParticipant`), fields (`TemplateField`), and split rules (`SplitRule` with `SplitRuleAllocation` percentages per participant).
- **TemplateInstance** → a runtime instance of a template (status: `IN_PROGRESS` / `SETTLED`). Creating an instance automatically creates `InstanceFieldValue` records for all template fields.
- **InstanceFieldValue** → a field entry with an amount, note, date, and a `split_mode`.
- **ParticipantEntryAmount** → the calculated amount owed by each participant for a given field value entry.

### Split Modes (`SplitMode` enum)
Three modes for how a field value's amount is split:
1. `TEMPLATE_FIELD_PERCENT_SPLIT` — uses the field's `default_split_rule_id`
2. `FIELD_VALUE_CUSTOM_PERCENT` — override with a custom `override_split_rule_id` on the field value
3. `FIELD_VALUE_FIXED_AMOUNTS` — direct fixed amounts per participant in `ParticipantEntryAmount`

### Frontend
React 18 / TypeScript SPA in `frontend/src/`. App name: **weeven**.

- **Stack** — React Router v6, TanStack Query (30s staleTime), Axios, Tailwind CSS, shadcn/ui components (Radix UI primitives in `components/ui/`). Path alias `@` maps to `src/`.
- **API layer** — `src/api/client.ts` wraps Axios and automatically unwraps `ApiResponse<T>`, throwing on `success=false`. Module files (`templates.ts`, `instances.ts`, `participantAmounts.ts`) expose typed functions consumed by hooks in `src/hooks/`.
- **Types** — `src/types/index.ts` mirrors all backend DTOs (flat UUID fields, not nested objects). Keep in sync with backend DTO changes.
- **Auth** — No authentication yet. `CURRENT_USER_ID` in `src/config/constants.ts` is a hardcoded UUID used for all API calls.
- **Routes** — `/templates`, `/instances`, `/instances/:id`, `/settled` — all nested under `AppShell`.

### Frontend UI Patterns

- **Theme** — Light mode only (no `class="dark"` on `<html>`). Custom CSS vars in `src/index.css`: light blue-white background (`--background: 224 60% 90%`), dark navy nav (`--nav: 222 72% 15%`), bright blue primary (`--primary: 222 91% 60%`), white cards (`--card: 0 0% 100%`).
- **Layout** — `AppShell` constrains content to `max-w-3xl` centered. `NavBar` uses `bg-nav` (dark navy) with brand name.
- **Participant color palette** — Participants are assigned colors by their index in the template's participant list. This palette is used consistently across `ParticipantTotalsBar` tiles and `FieldValueRow` pill badges:
  ```ts
  [blue-50/blue-200/blue-700, violet-50/violet-200/violet-700,
   emerald-50/emerald-200/emerald-700, orange-50/orange-200/orange-700]
  ```
- **Instance detail entry rows** (`FieldValueRow`) — Each row shows the dollar amount on the left (fixed `min-w-[5.5rem]`) and colored pill badges on the right (one per participant). Pills show `Name · split% · $amount` with distinct weights. Action buttons (edit/delete) use a fixed `w-[3.25rem]` container so pill alignment is consistent across SINGLE and MULTIPLE field types.

### API Documentation
Full API contracts are in `src/main/resources/api/docs/expense-splitter-api-contracts.md` and the OpenAPI spec is in `expense-splitter-openapi.yaml` in the same folder.

### Database Migrations
Flyway migrations live in `src/main/resources/db/migration/` (V1–V7). Add new migrations as `V{n}__description.sql`. The JPA `ddl-auto` is set to `validate`, so schema changes must go through Flyway.
